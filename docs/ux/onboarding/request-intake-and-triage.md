# Request Intake & Triage

This document defines how new client requests enter the system and how they are handled.

---

## Request Entry Point

Clients submit new requests via:
- “New task” or “Request work” actions
- Comments that indicate new work (manual triage)

All requests are treated as unprioritized until reviewed.

---

## Internal Handling

New requests land in an internal state such as:
- Requested
- Inbox
- Needs Review

(This state may or may not be visible to clients.)

---

## Provider Triage Options

For each request, the provider decides to:
1. Accept as a task
2. Ask a clarifying question
3. Convert to a new project
4. Decline or defer

No request implicitly changes scope or priority.

---

## When a Request Becomes a Project

A request should become a project if it introduces:
- New scope
- A new timeline
- Additional cost
- Increased risk

This ensures expectations remain clear.

---

## Summary

Requests are signals, not commands.

Triage is intentional and provider-owned.
