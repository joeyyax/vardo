# Offboarding & Hosting UX

This document defines the user experience for ending hosting or concluding a project.

Offboarding should feel calm, respectful, and controlled.

---

## Entry Point

Offboarding may begin when:
- Hosting is ended
- Project is formally concluded
- Client initiates a transition

Project state advances to **Offboarding**.

---

## Project Page — Offboarding Stage

### Status

> **Offboarding**  
> We’re wrapping things up and preparing your data.

### Visible Elements

- Request Application Data
- Migration Checklist
- Migration Assistance (optional)

No tasks or new work can be created.

---

## Request Application Data

### UX

- Clear button: **Request application data**
- Explains what is included:
  - Application code
  - Database backup
  - Media files
- Explains what is not included:
  - Billing records
  - Internal notes
  - Client management data

Status feedback:
- Requested
- Processing
- Ready

Exports appear in Project Docs when complete.

---

## Migration Checklist

### Purpose

Help clients self-serve a transition.

Checklist includes:
- Planning
- Data export
- New environment setup
- Validation
- Decommissioning

Checklist is informative, not blocking.

---

## Migration Assistance

Optional, clearly marked as paid.

Clients can:
- Request assistance
- View scope tiers
- Ask questions

No assumption of free help.

---

## Constraints

- Offboarding does not revoke data access
- Data exports remain available during offboarding
- No “dark patterns” or urgency messaging

---

## Summary

Offboarding:
- Preserves trust
- Avoids lock-in fear
- Makes transitions boring (in a good way)

A good exit is part of a good product.
