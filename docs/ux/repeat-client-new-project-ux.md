# Repeat Client – New Project UX

This document defines the experience of starting a new project for an existing client.

The goal is speed without skipping structure.

---

## Entry Point

- Client already exists
- User selects **Start new project** from the Client overview

---

## New Project Creation

### Required

- Project name
- Primary contact (pre-filled from client contacts)

### Optional

- Internal notes

No re-onboarding of the relationship occurs.

---

## Getting Started (Repeat Client)

### Status

> **Getting Started**  
> We’re kicking off a new project. The process will look familiar.

### UX Differences vs First-Time Client

- “How We’ll Work Together” is visible but marked as previously reviewed
- Contacts are reused
- Communication norms are assumed

Everything else proceeds normally.

---

## Proposal → Agreement → Onboarding

All phases still occur:
- Proposal
- Agreement
- Onboarding
- Active

However:
- Onboarding is delta-based
- Existing infrastructure may be reused
- Only new or changed inputs are requested

---

## Multiple Projects View

Clients may see multiple projects at once.

Example:

Projects:
- Website Maintenance — Ongoing
- New Feature Build — Onboarding

Each project:
- Has its own status
- Has its own docs
- Has its own tasks

Nothing bleeds between projects unless explicitly linked.

---

## Summary

Repeat clients:
- Move faster
- See familiar structure
- Still benefit from clarity and gates

Projects never skip phases — they just move through them quickly.
