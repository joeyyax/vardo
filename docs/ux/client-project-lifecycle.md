# Client & Project Lifecycle

This document describes how clients and projects move through the system, from initial intake to active work and beyond.

The core idea is simple:

- **Clients are long-lived**
- **Projects are repeatable journeys**
- Each project moves through clear phases
- Clients always know where they are and what comes next

This applies equally to first-time clients and repeat clients.

---

## Core Concepts

### Client
A Client represents the ongoing relationship.

- A client may have multiple projects over time
- Communication norms do not reset between projects
- “How We’ll Work Together” applies at the client level

### Project
A Project represents a specific engagement.

- Each project has its own lifecycle
- Each project has its own proposal, agreement, and onboarding
- Projects never skip phases, but may move through them quickly

---

## Project Phases (Canonical)

Each project exists in exactly one phase at a time.

Phases are visible to the client and drive:
- Dashboard status
- Available actions
- Which documents are shown
- Which emails are sent

---

### Phase 0 — Inquiry (Internal)

**State:** `inquiry`  
**Client-visible:** No

- Initial contact or lead
- Fit is evaluated
- No dashboard or project yet

---

### Phase 1 — Getting Started

**State:** `getting_started`

This is the first client-visible phase.

Purpose:
- Orientation
- Expectation-setting
- Calm entry into the process

Client sees:
- Project status: *Getting Started*
- “How We’ll Work Together”
- A visible path forward

Nothing is required from the client yet.

---

### Phase 2 — Proposal

**State:** `proposal`

Purpose:
- Align on scope and pricing
- Make a clear decision

Client sees:
- Proposal document
- One primary action: **Review proposal**

Proposal acceptance moves the project forward, but does not start work.

---

### Phase 3 — Agreement

**State:** `agreement`

Purpose:
- Formalize the engagement
- Lock in rules of engagement

Client sees:
- Service agreement
- Relevant addenda (hosting, SOW, etc.)
- One primary action: **Accept agreement**

This is the true gate before work begins.

---

### Phase 4 — Onboarding

**State:** `onboarding`

Purpose:
- Gather access
- Perform setup
- Prepare for execution

Client sees:
- Onboarding checklist
- Contextual access requests
- Clear explanation of why each item is needed

Onboarding is completed entirely within the app.

---

### Phase 5 — Active

**State:** `active`

Purpose:
- Execute the work

Client sees:
- Tasks
- Documents
- Discussions
- Progress updates

This is the primary working phase for most projects.

---

### Phase 6 — Ongoing (Optional)

**State:** `ongoing`

Used for:
- Retainers
- Maintenance
- Long-term support

Client sees:
- Ongoing activity
- Requests
- Capacity or hours (if applicable)

---

### Phase 7 — Offboarding (If Applicable)

**State:** `offboarding`

Purpose:
- Calm, structured exit
- Data access and migration

Client sees:
- Request Application Data
- Migration checklist
- Migration assistance (optional)

Offboarding should never feel abrupt or adversarial.

---

## First-Time Client Experience

For a new client:

1. Client account and first project are created
2. Project enters **Getting Started**
3. Client is introduced to:
   - How communication works
   - What the process looks like
4. Proposal → Agreement → Onboarding proceed sequentially
5. Active work begins only after onboarding completes

The emphasis is on orientation and trust before requests.

---

## Repeat Client Experience

For repeat clients, the relationship already exists — but each project still has structure.

### What Does Not Reset
- Client account
- Communication norms
- Familiarity with the process

### What Does Reset
- Project phase
- Proposal
- Agreement
- Onboarding (project-specific)

A repeat client is onboarding **a new project**, not re-onboarding the relationship.

---

### Getting Started (Repeat Client)

Repeat clients still see **Getting Started**, but in a lighter form.

Copy may acknowledge familiarity:
> “We’re kicking off a new project. The process will look familiar.”

No re-education is required.

---

### Onboarding for Repeat Clients

Onboarding is **delta-based**.

Instead of requesting everything again, the system:
- Reuses existing infrastructure where appropriate
- Requests only what is new or different
- Confirms assumptions instead of re-asking basics

Examples:
- Existing hosting → no hosting setup
- New domain → request DNS access
- New app → request fresh credentials

---

## Wizard Model (How This Feels in the UI)

This system functions as a wizard without feeling like one.

Key characteristics:
- One project phase at a time
- One primary action per phase
- Future phases are visible but locked
- Docs appear as a result of progress

Clients never “start a wizard” or “finish a wizard.”
They simply move forward.

---

## Documents as Byproducts

Documents are not uploaded manually by clients.

Instead:
- Proposals become project docs
- Agreements become project docs
- Addenda become project docs
- Checklists become project docs
- Exports become project docs

The Project Docs area acts as an audit trail:
> “What has been agreed to and what has happened.”

---

## Email Alignment

Every phase transition may trigger an email.

Emails:
- Match dashboard language
- Never introduce surprises
- Reinforce progress

Emails narrate the journey; the dashboard is the source of truth.

---

## Guiding Principles

- One project, one phase
- One phase, one primary action
- Never ask for access before trust
- Never start work before agreement
- Never lock clients in
- Always show what’s next

---

## Summary

This lifecycle model creates:
- Clarity for clients
- Safety for the provider
- A calm, professional experience
- A system that scales from one-off work to long-term relationships

It treats freelancing as a product, not a scramble.
