# Client Visibility Model

This document defines what clients can see and do inside the system.

The goal is to maximize transparency and collaboration **without transferring workflow control**.

---

## Core Principle

**Visibility ≠ control**

Clients should always know:
- What is happening
- What is blocked
- What is complete
- What needs their input

They should not be responsible for managing workflow state.

---

## What Clients Can See

Clients can see:
- Projects they are invited to
- Project status and phase
- Tasks and task status
- Comments and discussions
- Documents (proposals, contracts, shared files)
- Invoices and reports (as enabled)
- Activity related to their projects

Nothing important is hidden.

---

## What Clients Can Do

Clients can:
- Comment on tasks and documents
- Answer questions
- Upload files (where enabled)
- Submit new task requests
- Approve or reject proposals and contracts
- Complete onboarding checklist items
- Request their data at any time

Participation is encouraged.

---

## What Clients Cannot Do (By Design)

Clients cannot:
- Change task status
- Move tasks between columns
- Reorder task priority
- Start or stop work implicitly
- Change project phase
- Modify scope through task manipulation

Any change that affects scope, timing, or cost must be explicit.

---

## Why This Exists

Giving clients workflow control:
- Creates ambiguity
- Introduces accidental scope changes
- Increases coordination overhead
- Blurs ownership

This model preserves trust while keeping responsibility clear.

---

## Summary

Clients are collaborators and stakeholders — not workflow operators.

The system is designed to make progress visible without creating cognitive burden.
