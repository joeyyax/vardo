# Projects vs Tasks

This document defines the boundary between projects and tasks.

The goal is to keep scope, billing, and expectations clear without adding unnecessary overhead.

---

## Core Rule

**If work introduces new scope, risk, pricing, or timelines, it should be a project.  
If work is execution within agreed boundaries, it should be a task.**

This rule is intentionally simple and defensible.

---

## What a Project Represents

A project exists to define or change the rules of work.

Projects introduce or modify:
- Scope
- Pricing or billing structure
- Timeline or delivery expectations
- Risk
- Deliverables
- Access or permissions

Projects always have:
- A proposal
- An agreement
- A lifecycle
- A clear start and end (even for ongoing work)

---

## What a Task Represents

A task is work performed *inside* already-agreed rules.

Tasks:
- Do not change pricing
- Do not redefine scope
- Do not require a new agreement
- Are expected within the current engagement

Tasks live entirely inside a project.

---

## Common Examples

### Retainer / Ongoing Project
- Content updates → Task
- Bug fixes → Task
- Small improvements → Task

### New Feature or Initiative
- New checkout flow
- Major redesign
- New integration

→ New project, even if:
- Same client
- Same site
- Same codebase
- Same team

---

## Why This Boundary Matters

Blurring this line leads to:
- Scope creep
- Awkward billing conversations
- Misaligned expectations
- Stress for both sides

Making the boundary explicit:
- Reduces friction
- Makes intent visible
- Protects trust

---

## Summary

Projects define the rules.  
Tasks execute within them.

When in doubt, create a project.
