# Tunnel Provider Architecture

Issue: #528 — Expose NAT'd Vardo instances for webhooks and optional app access.

## Problem

Vardo instances behind NAT cannot receive inbound connections. This blocks two things:

1. **GitHub webhooks** — push events can't reach a homelab node to trigger deploys
2. **App exposure** — deployed apps are unreachable from the public internet

The mesh already connects peers over WireGuard, but traffic only flows peer-to-peer over the tunnel. There's no mechanism for a public node to accept external HTTP traffic and relay it to a private node's apps.

## Decision

Introduce a **tunnel provider** abstraction (ports and adapters pattern) with three initial adapters. The private node is always the authority on what gets exposed — the public relay node never unilaterally adds routes.

### Why ports and adapters

The core domain logic (deciding what to expose, validating requests, tracking state) stays the same regardless of which tunnel technology carries the traffic. A Cloudflare Tunnel user and a mesh relay user both need the same authorization flow, the same database records, and the same UI. Only the "last mile" differs — how the actual TCP/HTTP traffic gets from the internet to the private node.

This also keeps the door open for providers we haven't thought of yet without touching core logic.

## Provider Interface

```typescript
// lib/tunnel/types.ts

/** What kind of endpoint is being exposed. */
export type TunnelTarget =
  | { kind: "webhook"; path: string }           // e.g. /api/v1/webhooks/github
  | { kind: "app"; appId: string; domain: string; port: number };

/** Current state of a tunnel route. */
export type TunnelRouteStatus =
  | "pending"      // requested, not yet active
  | "active"       // traffic is flowing
  | "error"        // provider reported a problem
  | "revoked";     // intentionally torn down

/** Result of a provision/deprovision call. */
export type TunnelResult = {
  ok: boolean;
  publicUrl?: string;    // the externally-reachable URL
  error?: string;
  providerMeta?: Record<string, unknown>;  // provider-specific state (CF tunnel ID, etc.)
};

/**
 * Tunnel provider contract.
 *
 * Each adapter implements this interface. The core orchestrator calls these
 * methods — it never touches provider internals directly.
 */
export interface TunnelProvider {
  /** Human-readable provider name for UI and logs. */
  readonly name: string;

  /** Provider identifier stored in DB records. */
  readonly kind: "mesh-relay" | "cloudflare" | "pangolin";

  /**
   * Create or update a tunnel route for the given target.
   * Idempotent — calling twice with the same target is safe.
   */
  provision(target: TunnelTarget, peerId: string): Promise<TunnelResult>;

  /**
   * Remove a tunnel route.
   * Idempotent — calling on a non-existent route returns ok: true.
   */
  deprovision(routeId: string): Promise<TunnelResult>;

  /**
   * Check health of an active route.
   * Used by the heartbeat loop to update route status.
   */
  healthCheck(routeId: string): Promise<TunnelRouteStatus>;

  /**
   * List all routes this provider is currently managing.
   * Used for reconciliation on startup.
   */
  listRoutes(): Promise<Array<{ routeId: string; publicUrl: string; status: TunnelRouteStatus }>>;
}
```

### Provider registry

```typescript
// lib/tunnel/registry.ts

import type { TunnelProvider } from "./types";

const providers = new Map<string, TunnelProvider>();

export function registerProvider(provider: TunnelProvider): void {
  providers.set(provider.kind, provider);
}

export function getProvider(kind: string): TunnelProvider {
  const p = providers.get(kind);
  if (!p) throw new Error(`Unknown tunnel provider: ${kind}`);
  return p;
}

export function listProviders(): TunnelProvider[] {
  return Array.from(providers.values());
}
```

## Native Mesh Relay Provider

The default provider. Uses the existing WireGuard mesh + Traefik on a public node to relay traffic.

### How it works

The private (NAT'd) node asks a public peer to relay specific routes. The public peer writes a Traefik dynamic config file that routes `Host(domain)` to the private node's internal mesh IP + port.

**Key insight**: the WireGuard tunnel already carries TCP traffic between peers. Traefik on the public node just needs a route that points to `http://10.99.0.X:PORT` — the mesh handles the rest.

### Traefik dynamic config generation

The public relay node writes files to its `traefik_dynamic` volume (already mounted in docker-compose.yml at `/etc/traefik/dynamic`). Traefik watches this directory (`providers.file.watch=true` is already configured).

```yaml
# /etc/traefik/dynamic/tunnel-{routeId}.yml
# Auto-generated by Vardo mesh relay — do not edit manually.

http:
  routers:
    tunnel-{routeId}:
      rule: "Host(`{domain}`)"
      entryPoints:
        - websecure
      tls:
        certResolver: le
      service: tunnel-{routeId}

  services:
    tunnel-{routeId}:
      loadBalancer:
        servers:
          - url: "http://{peerInternalIp}:{port}"
```

For webhook-only routes, the router rule is more restrictive:

```yaml
      rule: "Host(`{relayDomain}`) && PathPrefix(`/api/v1/webhooks`)"
```

### Mesh relay provider implementation sketch

```typescript
// lib/tunnel/providers/mesh-relay.ts

import type { TunnelProvider, TunnelTarget, TunnelResult, TunnelRouteStatus } from "../types";
import { db } from "@/lib/db";
import { tunnelRoutes, meshPeers } from "@/lib/db/schema";
import { meshJsonFetch } from "@/lib/mesh/client";
import { eq } from "drizzle-orm";

export class MeshRelayProvider implements TunnelProvider {
  readonly name = "Mesh Relay";
  readonly kind = "mesh-relay" as const;

  async provision(target: TunnelTarget, peerId: string): Promise<TunnelResult> {
    // 1. Look up the relay peer (must be public — has a non-null endpoint)
    const relayPeer = await this.findPublicPeer();
    if (!relayPeer) {
      return { ok: false, error: "No public mesh peer available to relay traffic" };
    }

    // 2. Look up the requesting peer's internal IP
    const privatePeer = await db.query.meshPeers.findFirst({
      where: eq(meshPeers.id, peerId),
      columns: { internalIp: true },
    });
    if (!privatePeer) {
      return { ok: false, error: "Requesting peer not found" };
    }

    // 3. Send route request to the relay peer over the mesh
    const result = await meshJsonFetch<TunnelResult>(
      relayPeer.id,
      "/api/v1/mesh/tunnel/provision",
      {
        method: "POST",
        body: JSON.stringify({
          target,
          originIp: privatePeer.internalIp,
          requestingPeerId: peerId,
        }),
      }
    );

    return result;
  }

  async deprovision(routeId: string): Promise<TunnelResult> {
    const route = await db.query.tunnelRoutes.findFirst({
      where: eq(tunnelRoutes.id, routeId),
    });
    if (!route?.relayPeerId) return { ok: true };

    return meshJsonFetch<TunnelResult>(
      route.relayPeerId,
      "/api/v1/mesh/tunnel/deprovision",
      {
        method: "POST",
        body: JSON.stringify({ routeId }),
      }
    );
  }

  async healthCheck(routeId: string): Promise<TunnelRouteStatus> {
    const route = await db.query.tunnelRoutes.findFirst({
      where: eq(tunnelRoutes.id, routeId),
    });
    if (!route?.relayPeerId) return "error";

    try {
      const result = await meshJsonFetch<{ status: TunnelRouteStatus }>(
        route.relayPeerId,
        `/api/v1/mesh/tunnel/health/${routeId}`,
      );
      return result.status;
    } catch {
      return "error";
    }
  }

  async listRoutes() {
    const routes = await db.query.tunnelRoutes.findMany({
      where: eq(tunnelRoutes.providerKind, "mesh-relay"),
    });
    return routes.map((r) => ({
      routeId: r.id,
      publicUrl: r.publicUrl ?? "",
      status: r.status as TunnelRouteStatus,
    }));
  }

  /** Find the first online public peer (has a WireGuard endpoint). */
  private async findPublicPeer() {
    return db.query.meshPeers.findFirst({
      where: (p, { and, isNotNull, eq: e }) =>
        and(isNotNull(p.endpoint), e(p.status, "online")),
    });
  }
}
```

## New Mesh API Endpoints

These are peer-to-peer API routes, authenticated via `requireMeshPeer()` — the same pattern used by `/api/v1/mesh/heartbeat`.

### On the relay (public) node

```
POST /api/v1/mesh/tunnel/provision
```

Called by the private node's mesh relay provider. Validates the request, writes Traefik dynamic config, records the route locally.

Request body:
```json
{
  "target": { "kind": "webhook", "path": "/api/v1/webhooks/github" },
  "originIp": "10.99.0.3",
  "requestingPeerId": "abc123"
}
```

Response:
```json
{
  "ok": true,
  "publicUrl": "https://relay.example.com/api/v1/webhooks/github",
  "providerMeta": { "traefikFile": "tunnel-xyz.yml" }
}
```

---

```
POST /api/v1/mesh/tunnel/deprovision
```

Removes the Traefik dynamic config file and cleans up the local route record.

Request body:
```json
{ "routeId": "xyz789" }
```

---

```
GET /api/v1/mesh/tunnel/health/:routeId
```

Returns the current status of a tunnel route on this relay node.

---

### On the private (requesting) node

```
POST /api/v1/mesh/tunnel/request
```

Called by the private node's own UI/server action. This is the **user-facing** entry point — it picks the configured provider, calls `provision()`, and stores the result.

This is NOT a mesh-authenticated endpoint; it uses normal session auth.

```typescript
// Addition to lib/db/schema/mesh.ts

export const tunnelRoutes = pgTable(
  "tunnel_route",
  {
    id: text("id").primaryKey(),
    /** The peer that requested this tunnel (the NAT'd node). */
    requestingPeerId: text("requesting_peer_id")
      .notNull()
      .references(() => meshPeers.id, { onDelete: "cascade" }),
    /** The peer acting as relay (null for non-mesh providers). */
    relayPeerId: text("relay_peer_id")
      .references(() => meshPeers.id, { onDelete: "set null" }),
    /** Which provider is handling this route. */
    providerKind: tunnelProviderKindEnum("provider_kind").notNull(),
    /** What is being exposed. */
    targetKind: tunnelTargetKindEnum("target_kind").notNull(),
    /** For app targets: the app being exposed. */
    appId: text("app_id")
      .references(() => apps.id, { onDelete: "cascade" }),
    /** The domain/hostname assigned to this route. */
    domain: text("domain"),
    /** The port on the private node to forward to. */
    port: integer("port"),
    /** For webhook targets: the path prefix being exposed. */
    path: text("path"),
    /** The publicly-reachable URL once active. */
    publicUrl: text("public_url"),
    /** Current route status. */
    status: tunnelRouteStatusEnum("status").notNull().default("pending"),
    /** Provider-specific metadata (CF tunnel ID, Traefik filename, etc.) */
    providerMeta: jsonb("provider_meta").$type<Record<string, unknown>>(),
    /** Last time the health check confirmed this route was working. */
    lastCheckedAt: timestamp("last_checked_at"),
    /** Error message if status is "error". */
    lastError: text("last_error"),
    createdAt: timestamp("created_at").defaultNow().notNull(),
    updatedAt: timestamp("updated_at").defaultNow().notNull(),
  },
  (t) => [
    index("tunnel_route_requesting_peer_idx").on(t.requestingPeerId),
    index("tunnel_route_relay_peer_idx").on(t.relayPeerId),
    index("tunnel_route_app_idx").on(t.appId),
    unique("tunnel_route_app_uniq").on(t.appId, t.providerKind),
  ]
);
```

## Security Model

### Authorization

**Principle: the private node is the authority.** The relay node never unilaterally exposes routes. Every route must be explicitly requested by an authenticated mesh peer.

1. **Mesh peer auth** — All tunnel provision/deprovision requests between peers use the existing `requireMeshPeer()` Bearer token authentication (SHA-256 hashed, stored in `mesh_peer.token_hash`).

2. **User auth** — The user-facing "expose app" action uses normal session auth (Better Auth). Only org admins/owners can expose apps.

3. **Peer allowlist** — The relay node maintains a list of peer IDs that are allowed to request tunnel routes. This prevents a compromised peer from exposing arbitrary services.

### Validation

On the relay node, before writing Traefik config:

1. **Domain validation** — The requested domain must be a subdomain of the relay node's configured base domain, OR a domain the relay node has authority over. No arbitrary domain hijacking.

2. **Path validation** — For webhook targets, only `/api/v1/webhooks/*` paths are allowed. No exposing arbitrary internal endpoints.

3. **Port validation** — Only ports explicitly listed in the app's `exposedPorts` or `containerPort` are valid targets. The relay node verifies this against the bundle/config it received during project transfer.

4. **Route limits** — Each peer has a maximum number of concurrent tunnel routes (configurable, default 10). Prevents resource exhaustion on the relay.

### Rate limiting

- **Provision calls**: 10 per minute per peer (Redis-backed, using existing Redis instance)
- **Health checks**: Passive — runs on the heartbeat interval (already every 30s), no separate rate limit needed

### TLS

- For mesh-relay: Traefik handles TLS termination using the existing cert resolvers (Let's Encrypt, Google, ZeroSSL). The WireGuard tunnel between relay and private node is already encrypted.
- For Cloudflare: Cloudflare handles TLS edge termination. The tunnel to the origin is encrypted by cloudflared.
- For Pangolin/Newt: Depends on Pangolin's config — typically TLS at the edge, encrypted tunnel to origin.

### Secrets

- Cloudflare tunnel tokens stored encrypted in `tunnel_config.config` (using the same AES-256-GCM encryption used for `app.env_content`)
- Pangolin API keys: same treatment
- Mesh relay: no additional secrets — uses existing mesh token auth

### Revocation

When a mesh peer is removed (`meshPeers` row deleted):
- `tunnel_routes.requesting_peer_id` has `ON DELETE CASCADE` — all routes for that peer are deleted
- The relay node's reconciliation loop (runs on heartbeat) detects orphaned Traefik config files and removes them

## Migration Path

### For existing instances with a mesh

1. **Schema migration** — `pnpm db:push` adds the new tables and the `is_public` column on `mesh_peers`. No data migration needed since there are no existing tunnel routes.

2. **Detect public peers** — On first startup after update, a migration script checks each mesh peer: if `endpoint IS NOT NULL`, prompt the admin to mark it as `is_public` via the Settings > Mesh UI. We don't auto-mark because having a WireGuard endpoint doesn't guarantee ports 80/443 are exposed.

3. **Default provider** — If the instance has at least one public mesh peer, `tunnel_config` is seeded with `provider_kind: "mesh-relay", enabled: false`. The admin enables it when ready.

4. **Webhook auto-setup** — Once tunnel provider is enabled, the GitHub App webhook URL can be automatically updated to use the tunnel URL. This happens lazily when the next webhook would be needed (e.g., user enables auto-deploy on an app).

### For solo homelab users (no mesh peers)

1. After update, Settings > Mesh shows a "Tunnel Provider" section even without mesh peers.
2. User selects Cloudflare or Pangolin and enters credentials.
3. Webhook exposure works immediately.
4. App exposure is opt-in from the app networking tab.

### For public-only instances

No change needed. Tunnel provider section appears in settings but shows "Not needed — this instance is directly reachable." Detection: check if `VARDO_DOMAIN` resolves to a public IP that matches the instance's external IP.

## Trade-offs

| Decision | Gains | Gives up |
|----------|-------|----------|
| Private node as authority | Security — relay can't expose things unilaterally | Requires round-trip for every provision/deprovision |
| Traefik dynamic config files (not Docker labels) | Works without relay node restarting containers; hot-reload via file watch | Another thing to reconcile; files can drift from DB state |
| Webhook-only by default | Minimal attack surface; most users only need webhooks | Extra step to expose apps |
| One route per app per provider (unique constraint) | Simple mental model; no duplicate routes | Can't have same app exposed via mesh-relay AND Cloudflare simultaneously |
| `tunnel_config` as separate table (not `system_settings`) | Typed, queryable, supports multiple providers | Another table to maintain |

## Open Questions

1. **Subdomain allocation for app exposure** — Should the relay node auto-assign `{app-name}.{relay-domain}`, or should the user pick? Auto-assign is simpler but could collide across peers.

2. **Multi-relay** — If the mesh has multiple public peers, should the private node pick one or load-balance across them? Starting with "pick one" (first online public peer) and revisiting later.

3. **Cloudflare Tunnel adapter scope** — Should the Cloudflare adapter manage the tunnel lifecycle (create/delete tunnel via API) or expect the user to pre-create it? Pre-created is simpler and more predictable.
