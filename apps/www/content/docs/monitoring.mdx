Vardo ships a full observability stack as part of its production Docker Compose configuration. Container resource metrics are collected from cAdvisor, stored in Redis Time Series, and streamed to the UI over Server-Sent Events. Container logs flow through Promtail into Loki and are queryable from the app. No external services are required.

## Architecture Overview

```
Docker containers
       │
       ├─── cAdvisor ──────────────────────────────► Redis TimeSeries (7-day retention)
       │       │                                              │
       │       └── broadcast.ts (SSE broadcast) ────────────►│
       │                │                                     │
       │                └── /api/.../stats/stream ──► UI charts (real-time)
       │
       └─── Promtail ──────────────────────────────► Loki ──► /api/.../logs/stream ──► UI logs
```

### Production Services

All monitoring services run automatically in the `production` Docker Compose profile:

| Service | Image | Default Port | Role |
|---------|-------|-------------|------|
| `vardo-cadvisor` | `gcr.io/cadvisor/cadvisor:latest` | 7300 | Container resource metrics |
| `vardo-loki` | `grafana/loki:3.4` | 7400 | Log aggregation and storage |
| `vardo-promtail` | `grafana/promtail:3.4` | internal | Log collection from Docker |

These services communicate over the internal Docker network and are not exposed publicly by default.

## Container Metrics

### What is Collected

For each running container labeled with `host.managed=true`, Vardo collects:

| Metric | Description | Key |
|--------|-------------|-----|
| CPU % | Container CPU utilization | `metrics:{project}:cpu:{containerId}` |
| Memory usage | Bytes used by the container | `metrics:{project}:memory:{containerId}` |
| Memory limit | Container memory limit | `metrics:{project}:memoryLimit:{containerId}` |
| Network Rx | Bytes received | `metrics:{project}:networkRx:{containerId}` |
| Network Tx | Bytes transmitted | `metrics:{project}:networkTx:{containerId}` |
| Disk writes | Bytes written to disk | `metrics:{project}:diskWrite:{containerId}` |

Disk usage (total and per-project) is collected separately via `df` and Docker `system df`.

### Collection Schedule

The metrics collector (`lib/metrics/collector.ts`) uses a two-phase schedule:

- **Warmup phase** — first 20 ticks: every 5 seconds. Populates time series quickly after startup.
- **Normal phase** — after warmup: every 30 seconds. Steady-state collection.

This warmup avoids showing empty charts to users who open the UI immediately after Vardo starts.

### Storage: Redis TimeSeries

Metrics are stored using Redis Stack's time series commands (`TS.ADD`, `TS.RANGE`). Each metric gets its own key with:

- **Retention**: 7 days (`168h`)
- **Duplicate policy**: `LAST` (last value wins if two points have the same timestamp)
- **Labels**: `project`, `container`, `metric`, `organization` for cross-series queries

Time series keys are created lazily on first write. An in-process `Set` (`createdKeys`) avoids redundant `TS.CREATE` calls.

### Disk Write Alerts

Every 6th tick after warmup (~3 minutes at normal interval), the collector runs a disk write alert check (`lib/metrics/disk-write-alerts.ts`). This compares recent disk write rates against thresholds and emits `disk-write-alert` notifications when thresholds are exceeded.

## Real-Time Metrics Streaming (SSE)

Vardo uses Server-Sent Events to push live metrics to the browser without polling.

### SSE Endpoints

| Endpoint | Scope |
|----------|-------|
| `/api/v1/organizations/[orgId]/stats/stream` | All containers in an org |
| `/api/v1/organizations/[orgId]/projects/[projectId]/stats/stream` | All containers in a project |
| `/api/v1/organizations/[orgId]/apps/[appId]/stats/stream` | A single app's containers |

### Broadcast Pattern

`lib/metrics/broadcast.ts` implements a shared broadcast loop to avoid redundant cAdvisor polls when multiple browser tabs are open:

1. The first SSE subscriber starts the polling loop (5-second interval).
2. Each new subscriber receives the latest cached snapshot immediately on connect, then gets live updates.
3. When the last subscriber disconnects, polling stops.

This means cAdvisor is only queried as fast as the slowest subscriber requires, regardless of how many tabs are open.

### Snapshot Cache

`getLatestSnapshot()` returns the most recent metrics without waiting for the next poll. This is used by the health API endpoint to serve fast responses (~20ms) without hitting cAdvisor on every request.

## Historical Metrics

Historical data is queryable via:

| Endpoint | Description |
|----------|-------------|
| `/api/v1/organizations/[orgId]/stats/history` | Org-level history |
| `/api/v1/organizations/[orgId]/projects/[projectId]/stats/history` | Project history |
| `/api/v1/organizations/[orgId]/apps/[appId]/stats/history` | App history |

Results are bucketed and aggregated from Redis TimeSeries with configurable bucket sizes (default 5 minutes). The UI renders charts using Tremor with dedicated chart color tokens (`--metric-cpu`, `--metric-memory`, `--metric-network-rx`, `--metric-network-tx`, `--metric-disk`).

### Business Metrics

Vardo also tracks higher-level business metrics per org (deploy counts, success rates, backup totals) in separate TimeSeries keys (`metrics:business:{orgId}:{metric}`). These are displayed on the admin metrics tab.

## Log Aggregation (Loki + Promtail)

### How Logs Flow

1. **Promtail** mounts `/var/run/docker.sock` and `/var/lib/docker/containers` read-only.
2. It discovers containers via Docker socket service discovery, refreshing every 5 seconds.
3. Only containers with the label `host.managed=true` are scraped.
4. Logs are shipped to **Loki** at `http://loki:3100/loki/api/v1/push`.

### Labels Attached to Logs

Promtail extracts the following Docker labels and attaches them as Loki stream labels:

| Loki Label | Docker Label | Description |
|------------|-------------|-------------|
| `project` | `host.project` | Vardo project name |
| `project_id` | `host.project_id` | Vardo project UUID |
| `environment` | `host.environment` | Environment (production, staging) |
| `service` | `com.docker.compose.service` | Compose service name |
| `container` | container name | Docker container name |

These labels enable efficient log querying by project, environment, or container.

### Loki Configuration

Key settings in `config/loki.yml`:

| Setting | Value | Notes |
|---------|-------|-------|
| Retention | 168h (7 days) | Matches metrics retention |
| Max ingestion rate | 10 MB/s (burst 20 MB/s) | Per instance |
| Max query series | 500 | Prevents runaway queries |
| Storage | Filesystem (TSDB schema v13) | Local volume `loki_data` |
| Memory limit | 512 MB | Docker container cap |

### Log Streaming to UI

The endpoint `/api/v1/organizations/[orgId]/apps/[appId]/logs/stream` streams container logs to the browser over SSE by querying Loki in real time. The `LOKI_URL` environment variable configures the connection (`http://loki:3100` in production).

## System Health API

`GET /api/health` returns a fast system health snapshot including:

- CPU usage (aggregated across all containers)
- Memory usage (vs system total)
- Disk usage (via `df -B1 /var/lib/docker`)
- Per-resource status (`ok`, `warning`, `critical`)

Thresholds:

| Resource | Warning | Critical |
|----------|---------|----------|
| CPU | 80% | 95% |
| Memory | 85% | 95% |
| Disk | 80% | 90% |

The health endpoint uses the in-memory metrics snapshot (not a fresh cAdvisor call) so it responds in ~20ms.

## Notification System

Vardo's notification system delivers event alerts to configured channels. See also: [Admin guide](/docs/admin-guide) for configuring channels.

### Event Types

| Event | When Triggered |
|-------|---------------|
| `deploy-success` | App deployment completed successfully |
| `deploy-failed` | App deployment failed |
| `backup-success` | Backup job completed |
| `backup-failed` | One or more backup volumes failed |
| `cron-failed` | Scheduled cron job failed |
| `volume-drift` | Volume content has drifted from baseline |
| `disk-write-alert` | Container disk write rate exceeded threshold |
| `auto-rollback` | Automatic rollback triggered after failed deploy |
| `invitation-sent` | User invited to organization |
| `invitation-accepted` | Invitation was accepted |
| `system-alert-service` | A monitored service went down |
| `system-alert-disk` | System disk usage is critical |
| `system-alert-restart` | A container has restarted unexpectedly |
| `system-alert-cert` | TLS certificate expiry warning |
| `system-alert-update` | Vardo update available |
| `weekly-digest` | Weekly summary of deploys, backups, and cron failures |

### Notification Channels

| Channel Type | Description |
|-------------|-------------|
| `webhook` | HTTP POST to a URL (Slack, Discord, generic) |
| `email` | Email via the configured provider |

Channels are configured per organization under **Settings → Notifications**. Each channel can subscribe to all events or a specific subset.

### Dispatch Architecture

`lib/notifications/dispatch.ts` listens to the internal event bus (`lib/bus`). When an event is emitted:

1. Active channels for the org are loaded from the database.
2. Each channel's `subscribedEvents` filter is checked.
3. Matching channels receive the event via their transport (webhook or email).
4. Failures are enqueued for retry via `lib/notifications/retry.ts`.
5. All deliveries (success and failure) are logged to the `notification_logs` table.

### Weekly Digest

A scheduled job assembles a weekly summary including deploy counts (succeeded/failed), backup counts, cron failures, and disk write alerts. The digest is sent to all channels subscribed to `weekly-digest`.

## Feature Flags Affecting Monitoring

Metrics and logs are not gated by feature flags — they are always available when the underlying services (cAdvisor, Loki) are running. However:

- `FEATURE_METRICS` (env var) — legacy gate; the current code always returns `true` for `isMetricsEnabled()`.
- `FEATURE_LOGS` (env var) — similarly, always available when Loki is reachable.

## Setting Up Monitoring from Scratch

If you installed Vardo using `install.sh`, monitoring is already running. Verify with:

```bash
docker ps | grep -E '(cadvisor|loki|promtail)'
```

If you installed manually without the production profile:

```bash
cd /opt/vardo
COMPOSE_PROFILES=production docker compose up -d cadvisor loki promtail
```

Confirm cAdvisor is collecting data:

```bash
curl http://localhost:7300/api/v1.3/containers/ | jq '.subcontainers | length'
```

Confirm Loki is receiving logs:

```bash
curl 'http://localhost:7400/loki/api/v1/labels' | jq .
```

## Grafana Integration

Grafana is not included in Vardo's Docker Compose stack, but you can connect it to the bundled Loki instance for advanced log analysis.

Add a Loki data source in Grafana pointing to `http://<your-server>:7400` (or the internal address `http://loki:3100` if Grafana is on the same Docker network).

> Note: Exposing Loki on a public port is not recommended without authentication. Use Grafana's built-in auth or place Traefik in front of the Loki port.

## Troubleshooting

### No metrics appearing in the UI

1. Check cAdvisor is running: `docker ps | grep cadvisor`
2. Check the `CADVISOR_URL` env var is set to `http://cadvisor:8080`
3. Verify the collector is running: check app logs for `[collector] Starting metrics collection`
4. Check Redis is running and the `REDIS_URL` is correct

### No logs appearing for a container

1. Verify the container has the label `host.managed=true`
2. Check Promtail is running: `docker ps | grep promtail`
3. Check Promtail logs: `docker logs vardo-promtail`
4. Confirm `LOKI_URL` is set to `http://loki:3100` in the app's environment

### Historical charts show gaps

Gaps indicate the collector was not running during that period (restart, crash, or cAdvisor downtime). The collector is fail-safe — cAdvisor unavailability skips that tick without crashing.

### Notifications not delivering

1. Check the channel configuration under **Admin → Settings → Notifications**
2. Check `notification_logs` table for failure details
3. For webhooks: verify the URL is reachable from the Vardo container
4. For email: verify the email provider is configured and the test email succeeds
