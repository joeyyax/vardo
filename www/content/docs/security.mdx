This document covers Vardo's security architecture — authentication methods, rate limiting, encryption, webhook verification, and production hardening recommendations.

## Authentication

Vardo uses [Better Auth](https://www.better-auth.com/) as its authentication framework. Multiple sign-in methods are supported and can be enabled or disabled via feature flags and admin settings.

### Authentication Methods

| Method | Description | Requirements |
|--------|-------------|-------------|
| **Email + password** | Standard credentials. Minimum 8 characters. | None (enabled by default; can be disabled) |
| **GitHub OAuth** | Sign in with GitHub. Email auto-links accounts. | `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` |
| **Magic link** | Passwordless email sign-in. | Email provider configured (Mailpace or Resend) |
| **Passkeys (WebAuthn/FIDO2)** | Hardware security keys or biometric authentication. | None |
| **TOTP 2FA** | Time-based one-time passwords with backup codes. | User opts in per-account |

### Password Policy

- Minimum 8 characters (enforced by Better Auth).
- Passwords are hashed by Better Auth (bcrypt by default).
- The `passwordAuth` feature flag controls whether email/password sign-in is available. When disabled, users must authenticate via passkey, magic link, or GitHub OAuth.

### Two-Factor Authentication (TOTP)

2FA uses TOTP (RFC 6238) with the issuer name `Vardo`. Backup codes are generated at enrollment and allow one-time account recovery if the authenticator app is lost.

Users enable 2FA from their account settings. Admins cannot force 2FA on users at this time — it is opt-in per user.

### Passkeys (WebAuthn/FIDO2)

Passkey authentication follows the WebAuthn standard. Any FIDO2-compatible authenticator works: hardware keys (YubiKey, etc.), platform authenticators (Touch ID, Face ID, Windows Hello).

Passkey credentials are stored in the `passkey` table and scoped to the application origin. They cannot be used on a different domain.

### Session Management

Sessions are managed by Better Auth using the Drizzle database adapter:

- **Session duration**: 7 days.
- **Session refresh**: Sessions are updated every 24 hours on activity.
- **Storage**: Sessions are stored in the PostgreSQL `session` table.
- **Org context**: The active organization is tracked via the `host_current_org` cookie. On each request, membership is verified against the database — the cookie is not trusted on its own.

### Account Linking

GitHub accounts are automatically linked if the email matches an existing account. Google is also configured as a trusted provider for auto-linking. This allows users to sign in with OAuth after previously creating an account with email/password.

## Rate Limiting

Vardo implements a two-layer rate limiting architecture (documented in detail in `docs/adr/adr-002-rate-limiting-architecture.md`).

### Layer 1: Middleware (IP-based, in-memory)

`middleware.ts` applies a coarse per-IP limit to all `/api/` routes:

- **Limit**: 200 requests per minute per IP.
- **Implementation**: In-memory `Map` with a bounded size of 10,000 entries (LRU eviction).
- **Purpose**: Stops brute-force and obvious abuse before requests reach Redis or the database.

The in-memory approach is intentional. Middleware runs on every request; hitting Redis on every request adds unnecessary latency for a coarse "are you obviously abusing us" check. For a single-process self-hosted deployment, in-memory counters are correct.

### Layer 2: Route-level (Redis-backed, identity-aware)

Sensitive routes use `withRateLimit()` from `lib/api/with-rate-limit.ts`. This wraps individual route handlers with a Redis sliding window limiter.

#### Tiers

| Tier | Limit | Window | Identity | Used For |
|------|-------|--------|----------|---------|
| `auth` | 10 req | 60s | IP | Login, signup, magic link, passkey registration |
| `public` | 30 req | 60s | IP | Webhooks, mesh join |
| `mutation` | 60 req | 60s | User ID | Create, update, delete operations |
| `read` | 120 req | 60s | User ID | List, get, stream |
| `admin` | 30 req | 60s | User ID | Admin API endpoints |
| `critical` | 10 req | 60s | Org ID | Deploy, rollback |

#### Sliding Window Algorithm

The sliding window uses a Redis sorted set of request timestamps with a Lua script for atomicity:

1. Removes timestamps older than `now - windowMs`.
2. Counts remaining entries.
3. If under the limit, adds a new entry (using Redis server time for microsecond-precision member uniqueness).
4. Returns the count.

The `Retry-After` header is set to the number of seconds until the oldest request in the window ages out — more accurate than a fixed window reset time.

#### Fail-Open Behavior

If Redis is unavailable, all rate limit checks pass. A Redis outage does not take down the API. The middleware in-memory layer continues to operate during Redis downtime.

#### Better Auth Routes

The `/api/auth/[...all]` catch-all uses `withRateLimit()` on the POST export:

```typescript
export const POST = withRateLimit(_POST, { tier: "auth" });
```

This applies the 10-req/min IP budget to all auth mutations (sign-in, sign-up, magic link, passkey registration) in a single place.

## Encryption at Rest

Environment variables and other secrets stored in the database are encrypted using AES-256-GCM.

### Key Hierarchy

```
ENCRYPTION_MASTER_KEY (256-bit, hex or base64)
        │
        ├── HKDF(sha256, master, salt=orgId, info="host-env-encryption", 32 bytes)
        │         → Per-org key (used for app environment variables)
        │
        └── HKDF(sha256, master, salt="host-system-settings", info="host-system-encryption", 32 bytes)
                  → System key (used for system_settings: GitHub App, email, backup credentials)
```

Each encryption operation generates a random 12-byte IV. The ciphertext format is:

```
enc:v1:<iv_hex>:<ciphertext_hex>:<auth_tag_hex>
```

The `enc:v1:` prefix makes encrypted blobs unambiguous — no false-positives on plaintext that happens to contain colons.

### Generating a Master Key

```bash
openssl rand -hex 32
```

Set this as `ENCRYPTION_MASTER_KEY` in `.env.prod` or `vardo.secrets.yml`. The key must be consistent across restarts — changing it will make previously encrypted data unreadable.

### What is Encrypted

- App environment variables (per-org key)
- System settings stored in the `system_settings` table: GitHub App private key, webhook secret, email API keys, backup storage credentials (system key)

### Fallback Behavior

`decryptOrFallback()` and `decryptSystemOrFallback()` return the raw value if it is not in encrypted format. This supports rows written before encryption was added (unmigrated data) without breaking reads.

## GitHub Webhook Signature Verification

The webhook endpoint at `/api/v1/github/webhook` verifies every incoming request using HMAC-SHA256:

```
expected = "sha256=" + HMAC-SHA256(secret, request_body)
```

Comparison uses `timingSafeEqual` to prevent timing attacks. If `GITHUB_WEBHOOK_SECRET` is not configured, the endpoint returns 500 and refuses all requests — there is no insecure fallback.

The webhook secret is sourced from:
1. Database (`system_settings` → `github_app`) — preferred for setup-wizard deployments.
2. `GITHUB_WEBHOOK_SECRET` environment variable — fallback.

## CSRF Protection

Better Auth includes CSRF protection for its own endpoints. Vardo's API routes (`/api/v1/`) are protected by session cookie authentication — requests without a valid session are rejected before any state-changing operation occurs. The session is verified server-side on every request via `requireSession()` or `requireOrg()`.

## Security Headers

Traefik handles TLS termination. Vardo does not set explicit security headers in Next.js by default. For a hardened production deployment, add these headers via Traefik middleware or `next.config.ts`:

```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=()
Content-Security-Policy: default-src 'self'; ...
```

## Multi-Tenant Isolation

All database queries are scoped by `organization_id`. The `requireOrg()` helper (used in all `/api/v1/organizations/[orgId]/` routes) verifies that the authenticated user is a member of the requested organization before the handler runs. Cross-org data access is not possible through the API — an org ID in the URL that the user does not belong to returns 403.

The `isAppAdmin` flag on the user table grants access to the admin panel and admin API routes. Admin routes use `requireAdmin()` (in `lib/auth/admin.ts`) which checks this flag after session verification.

## Secret Management Best Practices

### vardo.yml and vardo.secrets.yml

Vardo supports a two-file configuration system:

- `vardo.yml` — non-secret settings (instance name, domain, feature flags). Safe to commit.
- `vardo.secrets.yml` — secrets (encryption key, auth secret, API keys). Must be `0600`, gitignored.

The config file takes priority over database settings, which takes priority over environment variables.

```yaml
# vardo.secrets.yml
encryptionKey: "<64-char hex>"
authSecret: "<random string>"
email:
  apiKey: "<mailpace or resend api key>"
backup:
  accessKey: "<s3 access key>"
  secretKey: "<s3 secret key>"
github:
  clientSecret: "<github app client secret>"
  privateKey: "<base64-encoded PEM>"
  webhookSecret: "<random string>"
```

### Environment Variables

For deployments using `.env.prod` directly, the required secret variables are:

| Variable | Description |
|----------|-------------|
| `ENCRYPTION_MASTER_KEY` | 256-bit hex key. Generate: `openssl rand -hex 32` |
| `BETTER_AUTH_SECRET` | Auth signing secret. Generate: `openssl rand -hex 32` |
| `GITHUB_WEBHOOK_SECRET` | HMAC secret for GitHub App webhooks |
| `GITHUB_CLIENT_SECRET` | GitHub OAuth app client secret |

Never commit `.env.prod` or `vardo.secrets.yml` to version control.

## Recommended Cloudflare Configuration

If Vardo is behind Cloudflare:

1. **Set SSL/TLS mode to "Full (strict)"** — prevents SSL stripping between Cloudflare and your server.
2. **Enable "Under Attack" mode** during incidents — Cloudflare adds a JS challenge before requests reach Vardo.
3. **Rate limiting rules** — Cloudflare's WAF can rate-limit at the edge before requests reach Layer 1. This is additive, not a replacement.
4. **Firewall rules** — Block requests to `/api/v1/github/webhook` except from GitHub's IP ranges.
5. **Set `CF-Connecting-IP` as the trusted IP header** — the app reads `x-forwarded-for` for rate limit identity; ensure Cloudflare's header is passed correctly.

> If using Cloudflare, set `NEXT_PUBLIC_BETTER_AUTH_URL` and `NEXT_PUBLIC_APP_URL` to the Cloudflare-fronted domain, not the origin.

## Production Security Hardening Checklist

- [ ] `ENCRYPTION_MASTER_KEY` is set to a 256-bit random hex value
- [ ] `BETTER_AUTH_SECRET` is set to a long random string
- [ ] `GITHUB_WEBHOOK_SECRET` is set (if GitHub App is configured)
- [ ] `vardo.secrets.yml` is `chmod 600` and gitignored
- [ ] Traefik is configured with a valid ACME email for Let's Encrypt
- [ ] Ports 7300 (cAdvisor), 7400 (Loki) are firewalled — not publicly accessible
- [ ] PostgreSQL and Redis are on the internal Docker network — not publicly accessible
- [ ] `passwordAuth` feature flag is set to `false` if you only want passkey/OAuth login
- [ ] Registration mode is set to `closed` or `approval` after initial setup
- [ ] Cloudflare or equivalent DDoS protection is in place
- [ ] Backups are configured and tested
- [ ] SSH access to the host is restricted (key-only, no password)

## Planned Features

### API token authentication

> **Planned** — Tracked in [#172](https://github.com/joeyyax/vardo/issues/172)

The API currently requires session cookies for authentication. `vardo_`-prefixed API tokens can be created in the dashboard, but Bearer token authentication is not yet wired into the request middleware. Until this ships, use session cookies for programmatic access (see [API Reference](/docs/api-reference)).

### Forced 2FA

Admins cannot currently require 2FA for all users. Two-factor authentication is opt-in per user. A future release will add an org-level setting to require 2FA before users can access organization resources.

### Audit log

A structured audit trail of admin and deployment actions is not yet implemented. All state-changing operations go through authenticated endpoints, but there is no queryable log of who did what and when.

### IP allowlisting

There is no built-in IP allowlist for the admin panel. Use a firewall rule or Cloudflare Access rule to restrict access to `/admin` by IP.
